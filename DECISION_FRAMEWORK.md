# 實作判斷決策框架

> 將個人判斷邏輯文件化，防止主觀决策導致功能遺漏
> 
> **建立日期**：2026-03-26
> 
> **基於問題**：將「語氣控制」誤判為「可選優化」

---

## 📋 決策框架

### Q1: 這是「要求」還是「建議」？

使用 `spec_intent_classifier.md` 判斷：

```
[ ] 包含「必須」「應該要」→ MUST → 100% 實現
[ ] 包含「建議」「推薦」「可加入」→ SHOULD → 預設實現
[ ] 包含「可以」「如需」→ MAY → 不實現（除非明確需求）
```

---

### Q2: 這是「缺失」還是「可選」？

使用以下矩陣判斷：

| 評估維度 | 致命缺失 | 重要缺失 | 可選優化 |
|----------|----------|----------|----------|
| **核心功能可用** | ❌ 否 | ✅ 是 | ✅ 是 |
| **輸出質量影響** | 大 | 中 | 小 |
| **錯誤處理影響** | 大 | 中 | 無 |
| **使用者體驗** | 明顯變差 | 輕微影響 | 無影響 |

**定義**：
- **致命缺失**：沒有這個功能，核心功能完全不可用
- **重要缺失**：功能可用但明顯變差
- **可選優化**：功能可用，無明顯影響

---

### Q3: 偏差理由是否充分？

當 SHOULD 不實現時，檢查：

```
[ ] 是否有時間限制？（緊急修復、期限緊迫）
[ ] 是否有技術障礙？（依賴未就緒、API 不可用）
[ ] 是否有替代方案？（功能已由其他方式實現）
[ ] 是否已記錄在 DECISIONS.md？
```

**判斷標準**：
- ✅ 理由充分：記錄在 DECISIONS.md，繼續
- ❌ 理由不充分：視為「重要缺失」，需實現

## 🔄 判斷流程圖

```
收到規格書描述
     │
     ▼
Q1: 這是「要求」还是「建議」？
     │
     ├── MUST → 直接實作 ✅
     │
     ├── SHOULD → 預設實作 ⚠️
     │               │
     │               ▼
     │         Q3: 偏差理由是否充分？
     │               │
     │               ├── ✅ 理由充分 → 記錄 DECISIONS.md，標記為「不實現」
     │               │
     │               └── ❌ 理由不充分 → 視為「重要缺失」，實作
     │
     └── MAY → 不實作（除非有明確需求）
```

## 📊 實際案例

### 案例 1：語氣控制（誤判）

**原始描述**：PDF P8 - 「建議加入停頓控制」

**錯誤判斷**：
- 判斷：這是「可選優化」
- 理由：覺得不是核心功能
- 結果：未實現 → 後來發現是「重要缺失」

**正確判斷**：
- 關鍵詞：「建議」→ SHOULD
- Q2 評估：影響輸出質量（中）→ 重要缺失
- 結論：應該預設實作

---

### 案例 2：熔斷器

**原始描述**：PDF 提及「生產環境需處理網路不穩定」

**判斷**：
- 關鍵詞：隱含建議 → SHOULD
- Q2 評估：網路問題導致完全失敗 → 重要缺失
- 結論：應該實作（即使不完整）

---

### 案例 3：WAV 輸出

**原始描述**：PDF P6 - 「若需匯出為 .wav，可在此整合 ffmpeg 指令」

**判斷**：
- 關鍵詞：「若需」→ MAY
- Q2 評估：MP3 已經可用 → 可選優化
- 結論：不實作（但後來選擇實作是因為容易）

## 📖 使用方式

### 開發流程整合

1. **規格書閱讀階段**
   - 使用 `spec_intent_classifier.md` 標記所有描述
   - 使用本框架進行 Q1 判斷

2. **實作規劃階段**
   - 對所有 SHOULD 進行 Q2 評估
   - 識別「重要缺失」

3. **偏差處理階段**
   - 對不實現的 SHOULD 進行 Q3 判斷
   - 理由不充分則實作

4. **Review 階段**
   - 檢查是否有 SHOULD 被錯誤標記為「不實現」
   - 檢查 DECISIONS.md 是否完整
