### P0-1：MCP Protocol 標準化

| 維度 | 內容 |
|------|------|
| **整合複雜度** | 低 |
| **架構衝擊** | 可漸進 |

#### Before（現況）

```
Agent/Tool 介面：
┌─────────────┐     自訂 REST API      ┌─────────────┐
│  cli.py     │ ◄───────────────────► │ tool_registry │
└─────────────┘     無標準格式          └─────────────┘

問題：
- 每個 tool 都要客製化 adapter
- 無法接入 MCP ecosystem（1000+ servers）
- OpenAI/Anthropic/Google tool schema 不通用
```

#### After（目標）

```
Agent/Tool 介面：
┌─────────────┐     MCP Protocol      ┌─────────────┐
│  Agent      │ ◄══════════════════► │ MCP Server  │
└─────────────┘     標準化 schema     └─────────────┘
                                              │
                                              ▼
                                    ┌─────────────────┐
                                    │ MCP Ecosystem   │
                                    │ (1000+ servers) │
                                    └─────────────────┘
```

#### 解決的問題

| 問題 | 說明 |
|------|------|
| **Vendor lock-in** | tool interface 擺脴特定 provider |
| **重複 adapter** | 每個新 tool 不用重寫 |
| **生態封閉** | 無法使用 MCP 社群的大量 servers |

#### 帶來的好處

| 效益 | 量化 |
|------|------|
| **新 tool 接入時間** | 從 ~2天 → ~2小時 |
| **可用的 tool 數量** | 從自建 10個 → MCP 生態 1000+ |
| **維護成本** | 降低 ~60%（標準化介面） |

#### Owner

- **Agent**：實作 `mcp_*.py` servers
- **Johnny**：定義 tool whitelist + review

### P0-2：HOTL/HITL/HOOTL 分級治理

| 維度 | 內容 |
|------|------|
| **整合複雜度** | 低 |
| **架構衝擊** | 可漸進 |

#### Before（現況）

```
治理模式：
所有專案 → 統一 HITL（人類必經決策點）

問題：
- 70% 簡單專案（如 config 改動）浪費人力
- 複雜專案缺乏足夠人類監管
- 人力瓶頸：每個 phase 都要 human approval
```

#### After（目標）

```
治理模式：
┌──────────────────────────────────────────────┐
│ 專案風險評估（8-維度）→ 動態分級           │
├──────────────────────────────────────────────┤
│ HOOTL (風險 0-3)：                          │
│  → 自動化執行，人類事後抽樣審計             │
│                                              │
│ HOTL (風險 4-6)：                           │
│  → 即時監看 dashboard，可中斷                │
│                                              │
│ HITL (風險 7-10)：                          │
│  → 必經決策點，需簽核                       │
└──────────────────────────────────────────────┘
```

#### 解決的問題

| 問題 | 說明 |
|------|------|
| **人力瓶頸** | 低風險決策不需要 human approval |
| **一刀切治理** | 所有專案用同一套 rule |
| **效率浪費** | 簡單專案浪費 70% 人力 |

#### 帶來的好處

| 效益 | 量化 |
|------|------|
| **人力介入頻率** | 降低 ~70%（低風險專案全自動）|
| **決策延遲** | 低風險從 ~4小時 → ~0（自動）|
| **開發速度** | 簡單專案週期縮短 ~50% |

#### Owner

- **Agent**：實作 `tiered_governance.py`
- **Johnny**：定義分級閾值 + 審查高風險決策

### P1-1：Kill-Switch 熔斷機制

| 維度 | 內容 |
|------|------|
| **整合複雜度** | 低 |
| **架構衝擊** | 可漸進 |

#### Before（現況）

```
失控處理：
┌─────────────┐
│ Agent 失控   │ → 無法立即停止 → 持續消耗資源
│ (loops/hangs)│ → 錯誤蔓延 → 需要手動重啟
└─────────────┘

問題：
- 無自動化熔斷機制
- 需要人工介入才能停止
- 失控時間可能持續數分鐘到數小時
```

#### After（目標）

```
失控處理：
┌─────────────┐     熔斷觸發       ┌─────────────────┐
│ Agent 失控   │ ───────────────► │ Kill-Switch     │
│ (loops/hangs)│                  │ (<5秒停止所有)  │
└─────────────┘                   └─────────────────┘
                                            │
                                            ▼
                                  ┌─────────────────┐
                                  │ 所有 Agent 停止  │
                                  │ + 審計日誌       │
                                  │ + 通知 on-call  │
                                  └─────────────────┘
```

#### 解決的問題

| 問題 | 說明 |
|------|------|
| **失控無阻斷** | 失控 agent 持續消耗資源 |
| **人工介入延遲** | 需要人為發現才能停止 |
| **錯誤蔓延** | 一個 agent 問題影響整個流程 |

#### 帶來的好處

| 效益 | 量化 |
|------|------|
| **失控持續時間** | 從數分鐘 → <5秒 |
| **資源損耗** | 降低 ~90%（快速停止）|
| **風險暴露** | 降低 ~95%（錯誤蔓延阻斷）|

#### Owner

- **Agent**：實作 `circuit_breaker.py`
- **Johnny**：定義熔斷閾值

### P1-2：Multi-Agent Debate（5-Agent）

| 維度 | 內容 |
|------|------|
| **整合複雜度** | 中 |
| **架構衝擊** | 可漸進 |

#### Before（現況）

```
驗證流程（2-Agent）：
┌─────────────┐
│ Planner     │ ──► SpecCritic ──► 通過
└─────────────┘
     ▲
     │ 同一體系，可能 collusion
     └─────────────────────────────
     
問題：
- Planner/Critic 可能達成「默契共識」
- 單一視角無法發現隱性問題
- 3輪後仍分歧 → 無法解決
```

#### After（目標）

```
驗證流程（5-Agent）：
┌─────────┐     ┌────────────┐
│ Planner │ ──►│ SpecCritic │
└────┬────┘     └─────┬──────┘
     │                  │
     ▼                  ▼
┌─────────────────────────────────┐
│  Devil's Advocate (DA)          │ ← 專門唱反調
│  強制挑戰每個核心決策            │
└────────────┬────────────────────┘
             │
             ▼
      ┌────────────┐
      │ Truth      │ ← 專門查事實
      │ Validator   │
      └──────┬─────┘
             │
             ▼
      ┌────────────┐     ┌─────────────┐
      │ Judge      │ ──► │ Consensus   │ (cosine > 0.85)
      │(不同model) │     │ 或 Escalate │
      └────────────┘     └─────────────┘
```

#### 解決的問題

| 問題 | 說明 |
|------|------|
| **Planner/Critic collusion** | DA 專門拆夥 |
| **單一視角盲點** | 5個不同角色覆蓋多元觀點 |
| **強制挑戰不足** | DA 必須提出 5-10 個挑戰 |

#### 帶來的好處

| 效益 | 量化 |
|------|------|
| **錯誤發現率** | 提升 ~40%（多視角）|
| **決策品質** | 3輪後共識率 > 85% |
| **幻觉/偷懶** | DA 強制暴露 |

#### Owner

- **Agent**：實作 `debate_engine.py` + 3個新 agents
- **Johnny**：定義 DA/Judge prompt + 審查爭議決策

### P1-3：UAF+CLAP+MetaQA 幻覺偵測

| 維度 | 內容 |
|------|------|
| **整合複雜度** | 中 |
| **架構衝擊** | 可漸進 |

#### Before（現況）

```
幻覺偵測：
┌─────────────┐
│ Agent 輸出  │ ──► 規則比對 ──► 通過/失敗
└─────────────┘

問題：
- 只能抓到「明顯錯誤」
- 無法偵測「模型自己不知道自己在胡說」
- 無法偵測「表面通順但事實錯誤」
```

#### After（目標）

```
三層幻覺偵測：

Layer 1 - UAF（不確定性估計）：
┌─────────────┐
│ Agent 輸出  │ ──► UAF Score ──► Token entropy
              │               Self-consistency
              │               External grounding
              │               (>0.5 = 疑似幻觉)

Layer 2 - CLAP（中間層探測）：
              ──► Attention Probe ──► 模型內部不確定信號

Layer 3 - MetaQA（定期回測）：
              ──► 每週 QA 重測 ──► 長期一致性追蹤
```

#### 解決的問題

| 問題 | 說明 |
|------|------|
| **「不知道自己不知道」** | UAF 抓 token-level entropy |
| **表面通順但錯誤** | Self-consistency 抓前後矛盾 |
| **長期一致性** | MetaQA 每週回測 |

#### 帶來的好處

| 效益 | 量化 |
|------|------|
| **幻觉發生率** | < 0.5%（目標）|
| **錯誤發現時間** | 即時（非事後）|
| **事實準確率** | 提升 ~30% |

#### Owner

- **Agent**：實作 `uaf.py` + `clap.py` + `metaqa.py`
- **Johnny**：定義閾值 + 審查高風險幻觉
