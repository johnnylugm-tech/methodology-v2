## 為什麼 constitution 版會「退化」？

### 1. 設計目標不同

constitution 版的設計目標是：
- **最小可行產品**：只保留核心功能
- **易於理解**：用於教學或概念驗證
- **依賴最小化**：減少維護負擔

這導致刻意「移除」了部分功能：
- 移除重試機制（覺得冗餘）
- 移除 WebSocket（覺得不需要）
- 移除 text_processor（覺得太複雜）

### 2. 過度精簡的反面

```
「簡化」的陷阱：
- 移除 text_processor → 無法載入 txt 檔案 → 直接 crash
- 移除重試機制 → 網路問題就直接失敗
- 移除錯誤處理 → 失敗也沒有日誌
```

### 3. 功能移植不完整

在「精簡化」過程中，把「應該有」的功能也一併移除了：

| 應該保留 | 為何被移除 | 後果 |
|----------|-----------|------|
| text_processor | 「太複雜」 | 無法載入 txt 檔案 |
| 重試機制 | 「用不到」 | 網路問題直接失敗 |
| 錯誤處理 | 「浪費程式碼」 | 失敗無訊息 |

## 教訓

### 1. 精簡化不等於移除核心功能

```
正確的精簡：
- 合併相似模組
- 移除重複程式碼
- 簡化介面

錯誤的精簡：
- 移除「覺得不需要」的功能
- 移除「太複雜」的模組
- 移除「浪費時間」的錯誤處理
```

### 2. 功能取捨要有原則

| 保留 | 移除 |
|------|------|
| 核心業務邏輯 | 裝飾性功能 |
| 錯誤處理 | 多餘的驗證 |
| 重試機制 | 未使用的選項 |
| 最低依賴 | 可以自己實現的套件 |

### 3. 版本演化要有基準測試

每次「精簡化」或「重構」後，必須確保：
- 基本功能仍然正常（能跑）
- 核心輸出仍然正確（音訊合併）
- 錯誤仍然被捕捉（日誌記錄）
