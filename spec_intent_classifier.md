# PDF 規格書意圖分類器

> 將 PDF 規格書中的描述分類為「要求」或「建議」，防止關鍵功能被遺漏
> 
> **建立日期**：2026-03-26
> 
> **基於問題**：將「語氣控制」誤判為「可選優化」

---

## 📋 分類定義

### 1. 必要要求（MUST）

**關鍵詞識別**：
- 「必須」
- 「應該要」
- 「要求」
- 「勢必」
- 語氣果斷的祈使句

**處理方式**：100% 實現，無理由不實現

**範例**：
```
❌ "系統必須支援多語系"
❌ "資料庫連線必須使用加密"
✅ 實現方式：直接實作
```

---

### 2. 強烈建議（SHOULD）

**關鍵詞識別**：
- 「建議」
- 「建議加入」
- 「推薦」
- 「可加入」
- 「建議使用」
- 「建議設計」
- 「建議配置」
- 「建議實現」

**處理方式**：
- **預設實現**
- 如不實現，必須記錄在 `DECISIONS.md` 並說明原因

**範例**：
```
❌ "建議加入停頓控制功能"
❌ "可加入錯誤重試機制"
✅ 預設實現：將其視為「要求」同等級
✅ 不實現需記錄："因時間緊迫，延後至 Phase 2"
```

---

### 3. 可選優化（MAY）

**關鍵詞識別**：
- 「可以」
- 「如需」
- 「例如」
- 「如圖所示」
- 「可選」
- 「可选实现」

**處理方式**：
- 不主動實現
- 除非明確需求再實作

**範例**：
```
✅ "可以加入自定義主題功能"
✅ "如需支援多租戶，可擴展架構"
```

---

### 4. 範例參考（REFERENCE）

**關鍵詞識別**：
- 「例如」
- 「如圖」
- 「示意圖」
- 「示意」

**處理方式**：
- 僅供參考，不一定要實現
- 可作為實作方向指引

---

## 🔄 語法檢查清單

當閱讀 PDF 規格書時：

```
[ ] 找到所有「必須」「應該」的描述 → 標記為 MUST
[ ] 找到所有「建議」「推薦」的描述 → 標記為 SHOULD（預設實現）
[ ] 找到所有「可以」「如需」的描述 → 標記為 MAY
[ ] 找到所有「例如」的描述 → 標記為 REFERENCE
[ ] 對於 SHOULD 如不實現 → 記錄在 DECISIONS.md
```

## 📊 判斷決策樹

```
這段描述是什麼類型？
    │
    ├── 包含「必須」「應該要」→ MUST → 100% 實現
    │
    ├── 包含「建議」「推薦」「可加入」→ SHOULD → 預設實現
    │                                              ↓
    │                                    不實現？→ 記錄 DECISIONS.md
    │
    ├── 包含「可以」「如需」→ MAY → 不實現（除非明確需求）
    │
    └── 包含「例如」「如圖」→ REFERENCE → 僅供參考
```

## 📖 使用範例

### 原始 PDF 描述（P8）

```
8. 語氣控制設計

建議加入停頓控制，逗號可設定為 200ms，句號為 500ms，
換行為 1000ms。例如：<break time="500ms"/>
```

### 分類結果

| 原文 | 關鍵詞 | 分類 | 處理 |
|------|--------|------|------|
| 建議加入停頓控制 | 建議 | **SHOULD** | 預設實現 |
| 逗號可設定為 200ms | 數值 | SHOULD 附屬 | 實現 |
| 例如：`<break time="500ms"/>` | 例如 | REFERENCE | 僅參考 |

### 實際操作

1. 將「停頓控制」加入實作清單
2. 不實現需記錄：`DECISIONS.md`
3. 實現後在 `SPEC_TRACKING.md` 標記為 ✅

## 🔧 整合進工作流程

### 1. 閱讀規格書階段

```
Step 1: 掃描全文，標記所有 SHOULD/MUST 關鍵詞
Step 2: 建立初始實作清單
Step 3: 對照 SPEC_TRACKING.md
Step 4: 預設所有 SHOULD 為「要實現」
```

### 2. Review 階段

```
Review 清單增加：
[ ] 是否有 SHOULD 被標記為「不實現」？
[ ] 不實現的理由是否記錄在 DECISIONS.md？
[ ] SPEC_TRACKING.md 是否完整？
```
